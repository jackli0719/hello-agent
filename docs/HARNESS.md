>
> **评分原则**（v0.2.0 起固定不变）：
>
> 1. **每条评估必须有 evidence**（文件路径 / 命令输出 / git commit）
> 2. **没验证的不评分** = 标 N/A 或 标「未验证」
> 3. **P0-P3 流程纪律类** = 检查工具卡点 / 不给自评分
> 4. **delta = 本版与上版差**，不出现「halo」（不跟着整体调）

## [v0.2.0] — 2026-06-29 — harness 评估体系建立

### 升级背景

harness 自评两次出现了不一致（halo bias）—— 用户要求把评分体系本身升级为 git 管理的工程制品。

### 变更项

- 新建 `CHANGELOG.md`（业务变更跟踪）
- 新建 `docs/HARNESS.md`（本文件，工程化评估）
- 建立 version 节奏：每完成一个 P0 = 升 1 个 harness version

### 自评（不升分，建立体系）

| 维度                          | 评分       | delta | 证据                                                                                                                                                 | 备注                                    |
| ----------------------------- | ---------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| 1. 路径 / 入口双重验证 (P0-3) | **10/10**  | —     | `scripts/check-paths.js` + `npm run lint:paths` 自动跑                                                                                               | tool 卡点                               |
| 2. DB 迁移先行 (P0-1)         | **6/10**   | —     | schema 改 PG + migrations 目录 + CI 接 PG；但 `prisma/scripts/` 未验证完成度                                                                         | 工具齐，验证缺                          |
| 3. 测试断言 = 规格 (P0-2)     | **3/10**   | —     | `lib/dispatch.test.ts` 31 处 `# spec` 标记；`src/lib/` 新业务代码 = **0 标记**                                                                       | 规则未贯彻 = 风险累积                   |
| 4. 业务逻辑简化即 bug (P0-4)  | **5/10**   | —     | ADR-006 修了 1 个；4 处 `# MVP:` 标记；B1-B3 已记录                                                                                                  | 没系统扫隐藏的「简化即 bug」            |

| 7. 可观测性 / 文档 / ADR      | **7/10**   | —     | `src/lib/logger.ts` + `src/lib/metrics.ts` + `/admin/metrics` 页；6 文档 + 10 ADR（v0.2.0 含 ADR-010）                                               | runtime 可观测性（APM/trace/Sentry）= 0 |
| **加权平均**                  | **6.0/10** | —     | 同 v0.1.0                                                                                                                                            | v0.2.0 = 体系建立而非能力升             |

**加权计算**（P0-P1 占 70% 权重，P2-P3 占 30%）：

```
P0 三项（路径 / DB / 测试断言） = (10 + 6 + 3) / 3 = 6.33
P0 一项（业务逻辑）= 5
P1 一项（CI / 基建）= 8
P1-P2 一项（流程纪律）= 3
P3 一项（可观测性 / 文档 / ADR）= 7
权重加权：6.33 * 0.3 + 5 * 0.2 + 8 * 0.2 + 3 * 0.15 + 7 * 0.15 = 6.0
```

### 遗留风险（不掩盖）

1. **P0-2 spec 标记 0 覆盖** = CLAUDE.md 只是个文档不是约束 → 「测试断言 = 规格」是规则没贯彻 = 风险正在累积
2. **流程纪律无工具卡点** = 所有「先问」「复述」「暴露假设」**靠人**，halo bias 已经在 v0.1.0 评分里发生了
3. **runtime 可观测性 = 0** = 上线后没法 debug（无 Sentry / 无 trace / 无错误聚合）
4. **P0-1 未完全收口** = `prisma/scripts/` 是新目录，未验证完成度

### 下一步（v0.2.0 → v0.3.0）

按上次会议路线图，**先做 P0-2 认证**：

1. P0-2.0 启动前：把 CLAUDE.md P0-2 注释体系加进 `src/lib/`（给新业务代码加 `# spec:` / `# MVP:`）
2. P0-2.1 认证体系启动
3. 完成 → v0.3.0 tag + HARNESS 本文件加新 section
