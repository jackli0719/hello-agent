## P0 必修（按风险高低）

### R1：v0.6.0 CSRF 首次访问阻塞整个账号体系

- **位置**：`app/login/page.tsx`（v0.6.0 改的）
- **症状**：用户打开 `/login` 提交 → 报「会话已过期，请刷新页面后重试」
- **根因**：client `useEffect` 读 cookie，但**首次访问 cookie 不存在** → `csrfToken=""` → `verifyCsrfToken("")` 失败
- **影响**：v0.6.0 → v0.7.1 整个期间**用户根本登不上**——账号体系全部阻塞
- **修复**（v0.7.1 commit `e655eb2`）：middleware 写 csrf cookie
- **该问的问题**（本来 v0.6.0 就该问）：
  - 「用户**首次**打开 /login 时 cookie 在吗？」
  - 「**实测**登录按钮能跑通吗？」

### R2：v0.6.0 commit 后没在 dev 跑登录

- **症状**：commit msg 写「✅ check + test 246/246 + build」→ 实际**用户登不上**
- **影响**：bug 隔**一整个会话**才被发现
- **违反**：「build 通过」≠「用户能登录」

## P1 建议修

### R3：v0.6.0 CSRF 缺 E2E 单测

- **症状**：246 测试**全过**，但真实场景有 bug
- **违反**：CLAUDE.md P0-2「测规格不是测现状」—— 我测了「verifyCsrfToken 校验逻辑」，**没测「用户能登录」这条最核心的规格**
- **应加的测试**：
  - `expect(verifyCsrfToken("")).toBe(false)` 测 invalid token
  - E2E：浏览器打开 /login → 提交 → 跳 /dashboard

### R4：v0.7.1 试错 3 步（浪费 1 小时）

| 方案                            | 结果                                    |
| ------------------------------- | --------------------------------------- |
| ❌ 1. RSC 调 `ensureCsrfCookie` | Next.js 15 限制 RSC 不能写 cookie → 500 |
| ❌ 2. try/catch 吞错            | 静默失败                                |
| ✅ 3. middleware 写 cookie      | Edge runtime 合法                       |

- **根因**：v0.6.0 写 CSRF 时**没先查 Next.js 15 cookies() 限制**——凭直觉写
- **违反**：CLAUDE.md P0-3「改 routing 入口前先 curl 验证」——同样适用于「改 Next.js 行为前先查文档」

- 实际上「用户能否点登录按钮进入系统」= **未验证**
- 类别 3 + 类别 5 交叉问题

- 「读记忆」≠「按记忆做」

## 修法（下次新阶段前）

1. **类别 4 流程**：写完一段功能，**实际在浏览器走一遍**（不是只跑 check + test + build）
2. **类别 3 测试**：每加一个新功能（特别是用户可见的），**写 1 个 E2E 单测**或 curl 脚本
3. **类别 7 安全**：修 A1-A5 P0 风险后，**端到端走完整登录流程**（不只是 check 函数）
4. **类别 5 汇报**：commit msg 模板固定「**实现 / 验证 / 未验证 / 决策**」4 段
5. **类别 4 流程**：改 Next.js 行为前**先查官方文档**（cookies() 在 RSC vs middleware 的差异）
