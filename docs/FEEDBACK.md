## 15. Agent 评审反馈记录（2026-06-30）

> 本节记录本轮 agent 对 harness 评审和整改过程中暴露的问题。它不是试用者模板的一部分，而是给后续 agent / 维护者看的过程反馈。

### 反馈 #A1：评分先行，文档未同步

- **问题**：先在对话中给出 8.7/8.8 的 harness 评分，但没有第一时间把评分依据写回 `docs/HARNESS.md`。
- **影响**：评分存在“对话里有效、仓库里不可追溯”的问题，违背 HARNESS 文件的证据化原则。
- **修正**：新增 `docs/HARNESS.md` v0.8.0 节点，记录 verify、test split、DB check、pre-commit、lib 白名单、SPEC、factory、smoke、文档去旧化等证据。
- **后续要求**：凡是对 harness 分数作实质调整，必须同步更新 `docs/HARNESS.md`，否则只能称为“临时口头评估”。

### 反馈 #A2：旧文档口径造成误导

- **问题**：README / DEPLOYMENT / ARCHITECTURE 仍残留 SQLite、`db:reset`、222 tests 等历史说法。
- **影响**：agent 读取文档时可能选择错误启动路径，尤其会误判当前数据库真值。
- **修正**：当前入口文档已统一为 PostgreSQL + `db:start` + `verify`；历史 SQLite 材料标为归档。
- **后续要求**：新增或迁移基础设施后，必须同步扫 README / DEPLOYMENT / ARCHITECTURE / `.env.example`。

### 反馈 #A3：测试环境失败噪音过大

- **问题**：此前 `npm run test` 在 PostgreSQL 不可达时会刷出大量 Prisma 连接失败。
- **影响**：agent 难以快速判断是环境问题还是业务断言失败。
- **修正**：新增 `scripts/check-test-db.js`，DB 不可达时快速失败并提示 `npm run db:start`。
- **后续要求**：所有需要外部依赖的测试入口都应先做前置检查，避免噪音掩盖真实失败。

### 反馈 #A4：页面级验证不足

- **问题**：此前 `verify` 只覆盖类型、格式、测试和构建，没有覆盖真实页面入口。
- **影响**：页面渲染、受保护路由重定向等 Next runtime 问题可能漏过。
- **修正**：新增 `scripts/smoke-pages.js` 并接入 `npm run verify`。
- **后续要求**：下一步应把 HTTP smoke 升级为真实交互 e2e，但需单独评估依赖和改动面。
