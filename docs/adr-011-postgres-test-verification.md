# ADR-011 — Postgres 迁移收口与真实验证（v0.2.3）

> **状态**：v0.2.3 节点写入 **HARNESS.md** 第 8 节 ADR 序列。
>
> **本文档**：补足 v0.1.0 / v0.2.1 / v0.2.2 决策回报里关于 Postgres 收口的「未验证」承诺。

---

## Context

v0.1.0 节点 P0-1 DB 迁移完成（schema 改 PG、migrations 建目录、CI 接 PG service container）。但 **v0.2.1 决策回报里我写**「`prisma/scripts/` 持续中——未验证完成度」、「`prisma/scripts/` 是新目录，未验证」——**意思是**：迁移脚本 + 验证脚本的存在 ≠ 在真实 Postgres 上跑通过。

v0.2.3 节点要做的事：**在真实跑过来的 PG 上验证测试全过 + 记录 PG 数据真正来源**。

## 决策

### 1. P0-1 真收口定义

P0-1 真收口 = **在 schema 配的 Postgres 上跑全测试 + 全过**。**不再要求**「PG 数据 = SQLite 数据」（这是次要问题）。

### 2. P0-3 实跑确认

- ✅ docker ps 显示 `o2o-pg-keepalive` 容器端口 5433 运行中
- ✅ Schema 已应用（5 张业务表 + `_prisma_migrations`）
- ✅ PG 端数据库有种子数据（5 cat / 8 sku / 5 master / 8 order / 2 rule）
- ✅ **222 测试在 PG 上全过**（v0.2.3 真跑 `npm run test`）

**这就是 v0.2.2 决策回报里做不到的事：真跑过测试。现在做到了。**

### 3. PG 数据的真实来源（决策回报主动暴露）

按 v0.2.3 跑的 `npm run db:verify-migration` 结果：

| 项              | SQLite (dev.db) | Postgres |
| --------------- | --------------- | -------- |
| ServiceCategory | 6               | 5        |
| ServiceSku      | 10              | 8        |
| Master          | 5               | 5        |
| Order           | 8               | 8        |
| DispatchRule    | 3               | 2        |

**id 完全不同**：PG 的 cuid 跟 SQLite 的 cuid 不一致。

**结论**：**PG 数据 ≠ SQLite 迁移来的**——PG 数据是**通过 `npm run db:seed` 直接生成的**（seed.ts 重新生成 cuid）。

**这不是 bug**——seed.ts 应该永远重新生成 cuid（避免硬编码种子），所以「id 不一致」是**预期行为**。

但**内容差异**（PG 缺 1 cat + 2 sku + 1 rule）说明**两边 seed 跑过的版本不同**——可能是某个时点更新 seed.ts 但只跑了一遍。**这是个真实的小问题**：

- 短期：不影响测试通过 + 不影响开发
- 中期：上线前 `npm run db:reset` 必须从 SQLite 重新跑迁移，或确保两边 seed 内容一致

### 4. `prisma/scripts/migrate-sqlite-to-postgres.ts` 状态

**v0.2.3 节点未实跑此脚本**——理由如下：

| 风险                                          | 等级  | 说明                                                                  |
| --------------------------------------------- | ----- | --------------------------------------------------------------------- |
| PG 已有数据（5 cat 等）+ 脚本是「存在则跳过」 | 🟡 P1 | 跑了会跳过 PG 现有数据，**不能补全 PG 缺失的 1 cat / 2 sku / 1 rule** |
| 跑了之后差异可能更大（cuid 全不同）           | 🟡 P1 | 跑了反而显得 PG 表「多」                                              |
| 上线前才需要跑（验证 SQLite 备份可恢复）      | 🟢 P2 | 不是 v0.2.3 范围                                                      |

**决策**：v0.2.3 节点**不实跑迁移脚本**，但脚本本身保留为「上线前的兜底工具」。**写 ADR-011 标记：脚本存在 = 可用工具，但未经实跑验证**。

### 5. `prisma/scripts/verify-migration.ts` 状态

**v0.2.3 节点实跑了**：

- ✅ 脚本能跑（不被 PR script bug 卡）
- ✅ 输出差异报告（说明工具有效）
- ✅ 报告有真实差异（说明它能捕获非一致情况）
- ⚠️ 报告的差异**不是迁移错**，是**两边 seed 数据 + 测试累积漂移**

**结论**：脚本**真有用**——上线时跑它能验证「SQLite 备份经脚本搬过来 vs PG 直接 seed 数据是否一致」。

### 7. P0-1 真收口后剩余风险

1. **种子内容双向不一致**（PG 缺 cat/sku/rule）——上线前 `npm run db:reset` 必须按统一节奏
2. **migrate-sqlite-to-postgres.ts 未实跑**——保留为兜底工具，**未验证**
3. **CI Postgres service** 实跑过没——这个本节也没真查（v0.2.1 决策回报里也是模糊）

## Consequences（按本 ADR 模型一致性）

- ✅ P0-1 真收口 = 222 测试在 PG 全过
- ⚠️ 脚本不等同卡点：迁移脚本没用过 = 不能拍胸脯说「迁移修复上线」
- ⚠️ 内容漂移是真实问题：下次 reset 必须验证两边 seed 一致
- ⚠️ ADR-011 自评涨幅有限：**单一维度涨不到 8 分**

## 补充节（v0.2.4 节点）：dev.db 的真相

> v0.2.3 节点写 ADR-011 时说「PG 数据 ≠ SQLite 迁移来的」+「内容差异」(PG 缺 1 cat / 2 sku / 1 rule) = 「真实的差异」。但**没分析根因**。v0.2.4 节点分析后：**根本不是 PG 数据问题**，而是 **dev.db 是历史快照**。

### 1. 数据来源真相

| 来源                     | 内容                                                      | 创建机制                                         |
| ------------------------ | --------------------------------------------------------- | ------------------------------------------------ |
| `lib/mock-data.ts`       | 8 个 SKU / 5 个 Master / 6 个 Order                       | **代码层真值**——修改这里生成新 seed              |
| `prisma/seed.ts`         | 引用 mock-data.ts 写入数据库                              | 创建数据**唯一渠道**（除手工操作外）             |
| `prisma/dev.db` (SQLite) | **10 个 SKU** / 5 Master / **8 个 Order** / **3 条 rule** | **历史快照**——版本在 mock-data.ts 修改之前       |
| 5433 上的 Postgres       | 8 个 SKU / 5 Master / 8 个 Order / 2 条 rule              | **通过 `npm run db:seed` 用当前 mock-data 生成** |

### 2. 差异逻辑推演

- **现在 mock-data.ts 是 8 个 SKU**——但 **dev.db 是 10 个 SKU**
- → **某个时点 mock-data.ts 删了 2 个 SKU**（S009 ZHUJIAO / S010 ZHONGJI-TAIQIU-ZHUJIAO）
- → **dev.db 没跟着跑 db:reset**——是历史快照
- → **PG 是当前 mock-data 跑 seed 生成**——所以 PG 没有 S009/S010

**所以 verify-migration 报的"差异"不是迁移脚本错，是 dev.db 是历史快照。**

### 3. 决策：保留 dev.db 作为历史快照

按用户确认（v0.2.4 节点拍板）：

- ✅ **保留 dev.db**——不再用 db:reset 重制（重制会丢失历史数据）
- ✅ **PG 是真值**（mock-data.ts + seed.ts = source of truth）
- ✅ **verify-migration 报告"不一致" = 正常的预期行为**——不视为 bug
- ❌ **不重写 dev.db**——避免数据丢失

### 4. verify-migration.ts 不需改造

之前的议题：

> 是否要把 verify-migration 改造为「schema 漂移」vs「历史差异」两类报告？

**决策：不改造**。理由：

- verify-migration 的职责 = 说「两边不一致」
- 由调用方 / ADR-011 解释**原因**
- 改脚本 = 改它本来的语义 = 副作用无止境
- 按「脚本职责单一」原则，verify 不该做诊断

### 6. 给 ARCHITECTURE.md 加的补充

需要更新 `docs/ARCHITECTURE.md` 第二节「技术栈选型」或「数据模型」章节，说明：

> **dev.db (SQLite) 的角色变化**：
>
> - **历史功能**：演示期数据持久化（v0.1.0 之前）
> - **当前功能**：作为**历史快照保留**——回滚兜底（如果 PG 不可用）
