# O2O MVP 下一阶段路线图（v0.9.2）

> **本路线图给试用者 + 维护者参考**。当前 MVP 在 Beta 阶段，验证「业务闭环跑通」；下面按优先级列下一阶段候选功能。
> 试用反馈见 [docs/FEEDBACK.md](FEEDBACK.md)，已知限制见 [docs/KNOWN_ISSUES.md](KNOWN_ISSUES.md)，验收清单见 [docs/BETA_CHECKLIST.md](BETA_CHECKLIST.md)。

---

## 分级原则

| 级别   | 含义                                                  | 处理时机       |
| ------ | ----------------------------------------------------- | -------------- |
| **P0** | **必须修复** — 核心流程阻断 / 业务规则缺失 / 安全漏洞 | Beta 期内      |
| **P1** | **影响试用和演示** — 试用者高频反馈 / 演示场景缺失    | 短期（≤ 1 月） |
| **P2** | **体验优化** — UI / UX / 性能 / 文档 / 开发者体验     | 中期（1-3 月） |
| **P3** | **未来规划** — 重大业务扩展 / 新商业模式 / 跨平台     | 长期（3 月+）  |

试用反馈中标记为 **P0** 的会进入「Beta 必修清单」，**P1** 进入下一阶段 sprint。

## P2 — 体验优化（中期）

| #    | 功能                                    | 业务背景                                                                                                                    | 预计工作量 |
| ---- | --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- | ---------- |
| P2-1 | **支付集成**（微信 / 支付宝）           | 试用者反馈：客户必须先付款后师傅才能上门。需要：订单创建后引导支付 → 支付回调改订单状态 → 未支付自动取消                    | 2-3 周     |
| P2-2 | **短信通知**（订单状态变化 → 客户手机） | 客户下单后需要确认 / 派单后通知 / 服务前 30 分钟提醒；当前缺第三方短信服务商集成（阿里云 / 腾讯云）                         | 1-2 周     |
| P2-3 | **地图定位 + 距离计算**                 | 派单推荐目前只看技能匹配，没考虑距离。需要：高德 / 百度地图 API + 师傅位置上报 + 服务范围 polygon                           | 2-3 周     |
| P2-4 | **订单导出**（CSV / Excel）             | 试用者（运营）反馈：财务对账需要导出订单明细。需要：列表页加「导出」按钮 + 后端生成 CSV                                     | 1 周       |
| P2-5 | **AI 派单优化**                         | 当前规则匹配是硬编码（requiredSkills superset + rating desc）；试用者反馈希望「综合评分 = 距离 + 评分 + 负载 + 历史完成率」 | 3-4 周     |
| P2-6 | **评价体系**（客户评价师傅）            | 客户完成后可以 1-5 星 + 文字评价；师傅评分从评价聚合                                                                        | 1-2 周     |
| P2-7 | **售后 / 工单系统**                     | 客户可发起「师傅态度差 / 服务不到位」投诉；客服介入处理；工单状态独立于订单                                                 | 2-3 周     |

## P3 — 未来规划（长期）

| #    | 功能                              | 业务背景                                                                                                                                    | 预计工作量 |
| ---- | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| P3-1 | **线上部署**（Vercel / 自建 K8s） | 当前默认本地 Docker PostgreSQL；试用者反馈「能在云上跑吗？」需要云数据库（Supabase / Neon / RDS）+ 部署流水线（CI/CD 推镜像）+ 域名 + HTTPS | 2-3 周     |
| P3-2 | **财务结算系统**                  | 师傅收入计算（订单金额 × 抽成比例）+ 月度账单 + 提现流程                                                                                    | 3-4 周     |
| P3-3 | **商家端（多租户 SaaS）**         | 当前是单租户（一家公司一个后台）；真实 O2O 平台需要支持多家服务商（家政公司 / 家电清洗公司）入驻 → 每个商家独立的师傅 / 服务 / 派单规则     | 4-6 周     |
| P3-4 | **师傅端原生 App**                | 当前是 H5（src/lib/worker.ts + app/worker/）；真实场景需要 React Native / Flutter App，集成推送 / 离线缓存 / 拍照上传                       | 6-8 周     |
| P3-5 | **数据看板 / BI**                 | 当前 `/admin/metrics` 只有基础计数器；真实业务需要：转化漏斗、师傅效率、SKU 热度、客户复购率                                                | 2-3 周     |
| P3-6 | **优惠券 / 营销工具**             | 新客立减 / 节日促销 / 分享返佣                                                                                                              | 2-3 周     |
| P3-7 | **AI 智能客服**                   | 试用者高频反馈「客户问的问题后台人工回复太慢」；需要 LLM + 知识库 + 多轮对话                                                                | 3-4 周     |

## 试用反馈决策机制

1. 试用者提交 [docs/FEEDBACK.md](FEEDBACK.md) 后，归类到 P0-P3
2. **P0** 进入「Beta 必修清单」（Sprint 0 必做）
3. **P1** 由维护者评审后进入下一 sprint
4. **P2 / P3** 进入候选 backlog，按 ROI（业务价值 / 工作量）排序

试用反馈越多越能帮助我们决定优先级，欢迎提交！
